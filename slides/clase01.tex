\begin{frame}{Objetivos de la clase}
\begin{itemize}
\item Reconocer qué vuelve distribuido a un problema y por qué la incertidumbre es estructural.
\item Razonar desde síntomas observables hacia posibles causas, sin confundir evidencia con certeza.
\item Documentar qué se relevó, qué hipótesis se analizaron y qué conclusión queda justificada.
\item Relevar, analizar y proponer en forma aislada y conectada.
\item Conectar el concepto con decisiones observables de diseño
\item Anticipar cómo reaparece en trabajos prácticos, parciales y sistemas reales
\end{itemize}
\end{frame}

\begin{frame}{Relevar, analizar, proponer}
\begin{columns}[T]
\column{0.32\textwidth}
\textbf{Relevar}
\begin{itemize}
\item síntomas observables
\item contexto y restricciones
\item datos que faltan
\end{itemize}

\column{0.32\textwidth}
\textbf{Analizar}
\begin{itemize}
\item causas compatibles
\item evidencia disponible
\item hipótesis descartadas
\end{itemize}

\column{0.32\textwidth}
\textbf{Proponer}
\begin{itemize}
\item garantía buscada
\item solución elegida
\item trade-off aceptado
\end{itemize}
\end{columns}

\vspace{0.5em}
\begin{uainfo}
Primero se separa cada dimensión para no confundir observación, inferencia y decisión.
Después se conectan para justificar una propuesta coherente.
\end{uainfo}
\end{frame}


\begin{frame}{Definición operativa}
Un sistema distribuido es un conjunto de nodos independientes que cooperan por red para ofrecer una abstracción unificada.

\begin{itemize}
\item No hay memoria compartida global
\item no hay reloj global confiable
\item los componentes pueden fallar de manera independiente
\item la red introduce demora, pérdida y reordenamiento
\end{itemize}
\end{frame}

\begin{frame}{Ejemplo motivador: pago con timeout}
Un cliente envía un pago.
El servidor procesa.
La respuesta no vuelve.

\begin{center}

  node distance=4cm,

]
\node[draw, rounded corners, minimum width=2.8cm, minimum height=1.2cm] (c) {Cliente};
\node[draw, rounded corners, minimum width=2.8cm, minimum height=1.2cm, right of=c] (s) {Servidor};
\draw[->, thick]
  (c.north east) -- (s.north west)
  node[midway, above=4pt] {pago};
\draw[->, thick, dashed]
  (s.south west) -- (c.south east)
  node[midway, below=6pt] {respuesta perdida / demorada};
\end{tikzpicture}
\end{center}

\begin{itemize}
\item ¿el servidor nunca recibió el pedido?
\item ¿procesó el pago y se perdió el ACK?
\item ¿está caído o solo lento?
\end{itemize}
\end{frame}

\begin{frame}{Insight central}
\begin{block}{Afirmación}
En un sistema distribuido asíncrono, un observador remoto no puede distinguir de manera perfecta entre una falla y una demora suficientemente larga.
\end{block}

Eso obliga a diseñar con hipótesis explícitas sobre fallas, garantías y recuperación.
\end{frame}

\begin{frame}{Tipos de fallas relevantes}
\begin{columns}[T]
\column{0.58\textwidth}
\begin{itemize}
\item crash: el nodo deja de responder
\item omission: un mensaje no llega
\item delay: un mensaje llega tarde
\item partition: una parte de la red queda aislada
\item byzantine: comportamiento arbitrario o malicioso
\end{itemize}

\begin{frame}{Formalización mínima}
Podemos representar una decisión de diseño como:
\[
D = (P,F,G,S,T)
\]
\begin{itemize}
\item $P$: problema
\item $F$: fallas asumidas
\item $G$: garantía requerida
\item $S$: solución elegida
\item $T$: trade-off aceptado
\end{itemize}
\end{frame}

\begin{frame}{Ejemplo usando el framework}
\textbf{Problema:} evitar duplicación de pagos.

\begin{itemize}
\item $F$: timeout y retry del cliente
\item $G$: a lo sumo un débito efectivo
\item $S$: idempotencia con request id único
\item $T$: más estado, más validaciones, más complejidad operativa
\end{itemize}
\end{frame}

\begin{frame}{CAP: qué significa ``bajo partición''}
Una partición ocurre cuando dos partes del sistema siguen vivas, pero no pueden comunicarse entre sí de manera confiable.

\begin{columns}[T]
\column{0.48\textwidth}
\begin{itemize}
\item los nodos no comparten una visión única del estado
\item cada lado puede recibir pedidos de clientes
\item no hay forma inmediata de coordinar una decisión global
\end{itemize}

\column{0.46\textwidth}
\begin{center}
\begin{tikzpicture}[node distance=1.5cm,every node/.style={font=\small}]
\node[draw, rounded corners, minimum width=2.2cm] (a) {Replica A};
\node[draw, rounded corners, minimum width=2.2cm, right=2.4cm of a] (b) {Replica B};
\node[below=0.75cm of a] (ca) {Cliente 1};
\node[below=0.75cm of b] (cb) {Cliente 2};
\draw[->] (ca)--(a);
\draw[->] (cb)--(b);
\draw[thick,dashed,red] (a)-- node[above] {sin enlace} (b);
\end{tikzpicture}
\end{center}
\end{columns}

\begin{uawarn}
CAP no dice que siempre hay que elegir dos letras. Dice que, si hay partición, responder en ambos lados puede sacrificar consistencia; coordinar o rechazar puede sacrificar disponibilidad.
\end{uawarn}
\end{frame}

\begin{frame}{CAP: intuición correcta}
\begin{columns}[T]
\column{0.54\textwidth}
Bajo partición, un sistema replicado no puede dar simultáneamente:
\begin{itemize}
\item consistencia fuerte observable
\item disponibilidad total de las operaciones
\end{itemize}

La decisión real es: rechazar para preservar corrección o responder sacrificando consistencia.
\column{0.40\textwidth}
\begin{center}
\begin{tikzpicture}[scale=0.95,every node/.style={font=\small}]
\node (C) at (0,0) {Consistencia};
\node (A) at (3.1,0) {Disponibilidad};
\node (P) at (1.55,2.4) {Particiones};
\draw[thick] (C)--(A)--(P)--(C);
\end{tikzpicture}
\end{center}
\end{columns}
\end{frame}

\begin{frame}{Errores conceptuales frecuentes}
\begin{itemize}
\item creer que timeout implica crash
\item pensar que la red es confiable salvo casos extremos
\item asumir que siempre existe una solución perfecta
\item usar CAP como slogan y no como restricción bajo partición
\end{itemize}
\end{frame}


\begin{frame}{Cierre}
\begin{block}{Idea clave}
La clase 01 agrega una nueva capa al pensamiento distribuido: no se trata de memorizar una técnica, sino de reconocer cuándo usarla, qué supone y qué sacrifica.
\end{block}
\end{frame}
