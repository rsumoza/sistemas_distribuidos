\UASectionSlide{Consenso y Raft}

\begin{frame}{Objetivos de la clase}
\begin{itemize}
\item entender por qué algunos problemas requieren consenso fuerte
\item distinguir safety y liveness
\item estudiar la estructura conceptual de Raft
\item analizar replicación de log y commit por mayoría
\end{itemize}
\end{frame}

\begin{frame}{Motivación}
Hay problemas donde no alcanza con “eventualmente converger”.

Ejemplos:
\begin{itemize}
\item elegir un líder
\item decidir una secuencia única de operaciones
\item comprometer un log replicado
\end{itemize}

\begin{block}{Pregunta}
¿Cómo garantizamos una sola historia aceptada por el sistema?
\end{block}
\end{frame}

\begin{frame}{Qué pide consenso}
Informalmente:
\begin{itemize}
\item los nodos deben acordar un mismo valor o una misma secuencia
\item no deben aceptar decisiones incompatibles
\item eventualmente, si las condiciones lo permiten, deben progresar
\end{itemize}
\end{frame}

\begin{frame}{Safety vs Liveness}
\textbf{Safety:}
\begin{itemize}
\item nada malo ocurre
\item por ejemplo: no hay dos líderes válidos que comprometan decisiones incompatibles
\end{itemize}

\textbf{Liveness:}
\begin{itemize}
\item algo bueno eventualmente ocurre
\item por ejemplo: se elige un líder y el sistema sigue procesando
\end{itemize}
\end{frame}

\begin{frame}{Estructura de Raft}
Roles:
\begin{itemize}
\item follower
\item candidate
\item leader
\end{itemize}

Idea central:
\begin{block}{}
Raft organiza el consenso alrededor de un líder que serializa las decisiones.
\end{block}
\end{frame}

\begin{frame}{Elección de líder}
\begin{itemize}
\item followers esperan heartbeats
\item si expira el timeout, un follower pasa a candidate
\item solicita votos
\item si obtiene mayoría, se convierte en leader
\end{itemize}
\end{frame}

\begin{frame}{Replicación de log}
El líder:
\begin{itemize}
\item recibe comandos del cliente
\item los agrega a su log
\item los envía a followers
\item compromete una entrada cuando una mayoría la confirmó
\end{itemize}
\end{frame}

\begin{frame}{Propiedad clave}
Una entrada comprometida no debe perderse en líderes posteriores.

Esta idea exige:
\begin{itemize}
\item reglas sobre votos
\item comparación de logs
\item restricciones sobre qué líder puede ganar
\end{itemize}
\end{frame}

\begin{frame}{Ejemplo tipo MIT}
Supongamos:
\begin{itemize}
\item un líder replica una entrada a un follower, pero no a mayoría
\item el líder cae
\end{itemize}

Pregunta:
\begin{block}{}
¿Puede el nuevo líder ignorar esa entrada?
\end{block}

Respuesta conceptual:
\begin{itemize}
\item si no fue committed, sí
\item si fue committed por mayoría, no
\end{itemize}
\end{frame}

\begin{frame}{Qué simplifica Raft}
Raft fue diseñado para ser más entendible que Paxos:
\begin{itemize}
\item liderazgo fuerte
\item separación clara entre elección y replicación
\item log único como objeto central
\end{itemize}
\end{frame}

\begin{frame}{Qué no resuelve Raft}
Raft no “elimina la red”:
\begin{itemize}
\item depende de timeouts
\item puede perder liveness bajo particiones severas
\item necesita mayoría conectada para progresar
\end{itemize}
\end{frame}

\begin{frame}{Errores frecuentes}
\begin{itemize}
\item creer que consenso es “solo votar”
\item confundir replicated state machine con almacenamiento replicado genérico
\item ignorar la diferencia entre una entrada replicada y una committed
\item pensar que Raft evita todo downtime
\end{itemize}
\end{frame}

\begin{frame}{Cierre}
\begin{block}{Idea clave}
Consenso fuerte aparece cuando la inconsistencia ya no es tolerable: el sistema necesita una historia común y una única decisión válida.
\end{block}
\end{frame}
