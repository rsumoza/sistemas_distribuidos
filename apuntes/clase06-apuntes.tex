\section{Introducción}

Los sistemas de almacenamiento constituyen la base sobre la cual se implementan las abstracciones de transacciones, consistencia y concurrencia. A diferencia de los modelos lógicos, aquí debemos enfrentar restricciones físicas reales:

\begin{itemize}
\item latencia de disco
\item costo de fsync
\item diferencia entre acceso secuencial y aleatorio
\end{itemize}

\begin{quote}
El diseño del storage engine define los límites reales del sistema.
\end{quote}

---

\section{Write-Ahead Logging (WAL)}

\subsection{Principio fundamental}

Antes de modificar cualquier dato persistente, el sistema debe registrar la operación en un log estable:

\[
\text{WAL write} \rightarrow \text{Data write}
\]

\subsection{Propiedad clave}

Esto garantiza:

\begin{itemize}
\item \textbf{Atomicidad}: operaciones incompletas pueden revertirse
\item \textbf{Durabilidad}: operaciones confirmadas pueden recuperarse
\end{itemize}

---

\subsection{Modelo de recuperación}

Sea un log:

\[
L = \{op_1, op_2, ..., op_n\}
\]

En caso de crash:

\begin{itemize}
\item \textbf{REDO}: reaplicar operaciones committed
\item \textbf{UNDO}: revertir operaciones no committed
\end{itemize}

---

\subsection{Problema sin WAL}

Si se escribe primero el dato:

\begin{itemize}
\item crash puede dejar estado inconsistente
\item no hay forma de reconstruir
\end{itemize}

---

\section{MVCC (Multi-Version Concurrency Control)}

\subsection{Idea central}

En lugar de sobrescribir datos:

\begin{quote}
cada escritura crea una nueva versión
\end{quote}

\subsection{Modelo formal}

Cada objeto:

\[
x = \{(v_1, t_1), (v_2, t_2), ...\}
\]

Una transacción con timestamp $t$ lee:

\[
\max \{v_i \mid t_i \le t\}
\]

---

\subsection{Ventaja principal}

\begin{itemize}
\item lectores no bloquean escritores
\item escritores no bloquean lectores
\end{itemize}

Esto elimina conflictos RW.

---

\subsection{Problemas}

\begin{itemize}
\item crecimiento de versiones
\item necesidad de garbage collection
\end{itemize}

---

\subsection{Garbage Collection}

Una versión puede eliminarse si:

\[
t_{version} < \min(\text{timestamps activos})
\]

---

\section{LSM Trees}

\subsection{Motivación}

Discos favorecen escrituras secuenciales:

\[
\text{secuencial} \gg \text{aleatoria}
\]

---

\subsection{Arquitectura}

\begin{itemize}
\item WAL (durabilidad)
\item Memtable (estructura en memoria)
\item SSTables (archivos inmutables en disco)
\end{itemize}

---

\subsection{Flujo de escritura}

\begin{enumerate}
\item escribir en WAL
\item insertar en Memtable
\item flush a SSTable
\item compaction
\end{enumerate}

---

\subsection{Compaction}

Proceso de merge:

\begin{itemize}
\item combina múltiples SSTables
\item elimina versiones antiguas
\item reduce fragmentación
\end{itemize}

---

\subsection{Problemas}

\begin{itemize}
\item write amplification
\item read amplification
\end{itemize}

---

\section{Índices}

\subsection{B-Tree}

\begin{itemize}
\item estructura balanceada
\item acceso en $O(\log n)$
\end{itemize}

---

\subsection{Comparación}

\begin{itemize}
\item LSM: alto throughput de escritura
\item B-Tree: baja latencia de lectura
\end{itemize}

---

\section{Integración}

Un storage engine moderno combina:

\begin{itemize}
\item WAL → durabilidad
\item MVCC → concurrencia
\item LSM/B-Tree → organización física
\end{itemize}

---

\section{Conclusión}

\begin{quote}
El almacenamiento no es una implementación: es la materialización del modelo de consistencia.
\end{quote}
