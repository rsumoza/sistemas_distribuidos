\section{Propósito de la clase}

La clase 2 introduce una idea central de toda la materia: en un sistema distribuido, toda coordinación entre componentes depende de mensajes que atraviesan una red no confiable. Esto significa que una operación remota nunca tiene la simplicidad semántica de una llamada local. Aun cuando la sintaxis se parezca a una invocación ordinaria, la semántica real está condicionada por demoras, pérdidas, reintentos, duplicaciones y fallas parciales.

La tesis principal de esta clase es la siguiente:

\begin{uakeybox}
Toda comunicación distribuida introduce incertidumbre sobre el estado remoto, y toda estrategia para reducir esa incertidumbre introduce a su vez nuevos trade-offs.
\end{uakeybox}

En consecuencia, no basta con aprender términos como \emph{timeout}, \emph{retry} o \emph{idempotencia}. Es necesario entender qué problema intenta resolver cada mecanismo, qué nuevas dificultades introduce y bajo qué supuestos puede considerarse correcto.

\section{De llamada local a llamada remota}

En un programa secuencial o concurrente dentro de una misma máquina, una llamada a función tiene propiedades intuitivas fuertes:

\begin{itemize}
\item el programador asume una pila de ejecución consistente;
\item si la llamada retorna, el control vuelve al punto esperado;
\item si la llamada falla, en general la falla es localmente observable;
\item no existe una red entre invocante e invocado.
\end{itemize}

En cambio, una llamada remota (\emph{Remote Procedure Call}, RPC) involucra al menos las siguientes etapas:

\begin{enumerate}
\item serialización de parámetros;
\item envío del mensaje por la red;
\item recepción por el servidor;
\item ejecución de la operación;
\item serialización de la respuesta;
\item envío de la respuesta;
\item recepción y deserialización por el cliente.
\end{enumerate}

Cada una de estas etapas puede fallar parcial o temporalmente. Por lo tanto, la abstracción ``llamada remota como si fuera local'' es útil pedagógicamente, pero peligrosa si se toma literalmente.

\section{No-inmediatez y capas de comunicación}

Una comunicación distribuida no debe entenderse como un salto instantáneo entre cliente y servidor. Entre la intención de invocar una operación y la observación de una respuesta hay una trayectoria compuesta por capas: aplicación, librería RPC o cliente HTTP, serialización, transporte, red, balanceadores, servidor, almacenamiento y camino de regreso.

Cada capa puede tener su propia forma de reaccionar ante una falla:
\begin{itemize}
\item la aplicación puede abandonar la espera y mostrar error;
\item el cliente RPC puede reintentar automáticamente;
\item TCP puede retransmitir segmentos sin que la aplicación lo observe directamente;
\item un balanceador puede redirigir a otra instancia;
\item el servidor puede seguir procesando aunque el cliente ya haya vencido su timeout;
\item la base de datos puede confirmar o abortar una operación de manera independiente de la respuesta de red.
\end{itemize}

Esto introduce una forma práctica de concurrencia: mientras una capa decide que ``ya esperó suficiente'', otra capa puede seguir intentando, entregando, ejecutando o confirmando efectos. Por eso, para diagnosticar un problema de comunicación no alcanza con preguntar ``falló o no falló''. Hay que reconstruir qué observó cada capa, qué acción tomó y qué efectos pudo producir.

\begin{uakeybox}
La no-inmediatez de la comunicación obliga a razonar con líneas de tiempo: lo que el cliente observa, lo que la red transporta, lo que el servidor ejecuta y lo que el almacenamiento confirma pueden no coincidir en el mismo instante.
\end{uakeybox}

\subsection{Error conceptual fundamental}

Uno de los errores más comunes en sistemas distribuidos es razonar así:

\begin{quote}
``Si llamo a un servicio y no obtengo respuesta, entonces el servicio falló.''
\end{quote}

Ese razonamiento es incorrecto. La falta de respuesta puede deberse a múltiples causas:

\begin{itemize}
\item el request nunca salió del cliente;
\item salió pero se perdió en la red;
\item llegó al servidor pero no se ejecutó;
\item se ejecutó pero la respuesta se perdió;
\item se ejecutó y la respuesta está demorada;
\item el servidor está vivo pero sobrecargado;
\item el servidor efectivamente cayó.
\end{itemize}

\begin{uawarn}
La observación ``no recibí respuesta'' no identifica la causa. Solo informa una ausencia de evidencia dentro de una ventana temporal determinada.
\end{uawarn}

\section{RPC como abstracción y como problema}

RPC busca ofrecer una interfaz procedural sobre una operación remota. El objetivo histórico fue simplificar la programación distribuida ocultando detalles de red. Sin embargo, desde el punto de vista conceptual, lo importante es entender que RPC no elimina el problema distribuido; solo lo encapsula.

Formalmente, una operación RPC puede representarse como una tupla:

\[
\mathcal{R} = (req, net_{out}, exec, net_{in}, resp)
\]

donde:

\begin{itemize}
\item $req$ representa la solicitud emitida por el cliente;
\item $net_{out}$ representa el trayecto de ida por la red;
\item $exec$ representa la ejecución en el servidor;
\item $net_{in}$ representa el trayecto de vuelta;
\item $resp$ representa la respuesta observada por el cliente.
\end{itemize}

El problema es que el cliente no observa directamente cada componente de la tupla. Solo observa, a lo sumo, el hecho de recibir o no una respuesta antes de que expire una política de espera.

\section{Timeout}

Un \emph{timeout} es una política de observación, no una prueba causal. El cliente fija un umbral temporal $\tau$ y decide:

\[
\text{si no llega respuesta antes de } \tau, \text{ entonces actuar}
\]

La acción puede ser:

\begin{itemize}
\item reintentar;
\item fallar hacia arriba;
\item degradar funcionalidad;
\item usar un valor alternativo;
\item redirigir a otra réplica o servicio.
\end{itemize}

\subsection{Interpretación correcta de un timeout}

Sea $T$ el tiempo real de finalización de una operación remota y $\tau$ el timeout configurado. Entonces:

\[
T > \tau
\]

solo implica que la operación no fue observada por el cliente dentro de la ventana de espera. No implica:

\begin{itemize}
\item que la operación no haya comenzado;
\item que la operación no haya terminado;
\item que el servidor haya fallado;
\item que el request no vaya a producir efectos más adelante.
\end{itemize}

\begin{uakeybox}
Un timeout divide el mundo entre ``observable a tiempo'' y ``no observable a tiempo'', no entre ``correcto'' y ``fallido''.
\end{uakeybox}

\subsection{Trade-off del timeout}

Timeouts cortos:
\begin{itemize}
\item mejoran capacidad de reacción;
\item reducen espera percibida;
\item aumentan falsos positivos de falla.
\end{itemize}

Timeouts largos:
\begin{itemize}
\item reducen reintentos innecesarios;
\item empeoran experiencia de usuario;
\item pueden retrasar detección de fallas reales.
\end{itemize}

No existe un timeout universalmente óptimo. El valor correcto depende de:

\begin{itemize}
\item latencia esperada del sistema;
\item criticidad de la operación;
\item costo de reintento;
\item tolerancia a demora;
\item perfil de carga;
\item modelo de fallas asumido.
\end{itemize}

\section{Retry}

Dado un timeout, una reacción natural es reintentar. Si una operación puede no haber ocurrido, repetirla parece razonable. Sin embargo, el retry no solo recupera de ciertas fallas; también puede amplificar problemas.

\subsection{Beneficio del retry}

El retry es útil cuando la falla es transitoria:

\begin{itemize}
\item pérdida ocasional de paquetes;
\item congestión momentánea;
\item reinicio rápido de un servicio;
\item failover entre instancias.
\end{itemize}

En estos casos, repetir puede aumentar significativamente la probabilidad de éxito.

\subsection{Riesgo del retry}

Si la operación original sí ocurrió y el cliente no lo sabe, entonces reintentar puede:

\begin{itemize}
\item duplicar pagos;
\item generar doble creación de recursos;
\item aplicar dos veces una mutación;
\item producir inconsistencia observable.
\end{itemize}

Además, retries masivos pueden generar \emph{retry storms}: la infraestructura, ya degradada, recibe todavía más tráfico debido a los propios mecanismos de recuperación.

\subsection{Retry y modelo de fallas}

El retry es correcto solo si se razona junto con:

\begin{itemize}
\item la naturaleza del efecto remoto;
\item la posibilidad de duplicación;
\item la capacidad del servidor para reconocer repeticiones;
\item la idempotencia de la operación;
\item una política de backoff.
\end{itemize}

\section{Idempotencia}

Una operación es idempotente si aplicarla varias veces produce el mismo efecto observable que aplicarla una sola vez.

Formalmente, si $f$ representa una operación sobre el estado $x$, entonces $f$ es idempotente si:

\[
f(f(x)) = f(x)
\]

\subsection{Ejemplos}

Ejemplos típicos de idempotencia:

\begin{itemize}
\item \texttt{setBalance(100)}: repetir la asignación sigue dejando saldo en 100;
\item \texttt{markAsRead(msg42)}: repetir no cambia el resultado final;
\item \texttt{PUT /resource/id} con el mismo contenido.
\end{itemize}

Ejemplos típicamente no idempotentes:

\begin{itemize}
\item \texttt{incrementBalance(100)};
\item \texttt{chargeCard(100)};
\item \texttt{createOrder()} sin clave de deduplicación.
\end{itemize}

\subsection{Idempotencia de protocolo vs idempotencia semántica}

Es importante distinguir:

\begin{itemize}
\item \textbf{idempotencia semántica}: la operación por su significado no cambia al repetirse;
\item \textbf{idempotencia implementada}: la operación no es naturalmente idempotente, pero el sistema la vuelve idempotente mediante IDs únicos, deduplicación o almacenamiento de resultados previos.
\end{itemize}

Por ejemplo, cobrar una tarjeta no es semánticamente idempotente. Sin embargo, puede implementarse de forma idempotente si toda operación lógica lleva una clave única y el servidor reconoce reintentos del mismo pedido.

\section{Semánticas de entrega}

En comunicación distribuida se suele hablar de tres familias de garantías.

\subsection{At-most-once}

La operación se ejecuta a lo sumo una vez.

Ventaja:
\begin{itemize}
\item evita duplicación.
\end{itemize}

Costo:
\begin{itemize}
\item si se pierde el request o la respuesta, puede perderse también la operación sin recuperación transparente.
\end{itemize}

\subsection{At-least-once}

La operación se reintenta hasta asegurar ejecución, aceptando la posibilidad de duplicados.

Ventaja:
\begin{itemize}
\item reduce riesgo de no ejecución.
\end{itemize}

Costo:
\begin{itemize}
\item exige idempotencia o reconciliación.
\end{itemize}

\subsection{Exactly-once}

La operación ocurre exactamente una vez desde el punto de vista lógico.

Esta semántica es la más deseada, pero también la más difícil. En la práctica, suele implementarse como una combinación de:

\begin{itemize}
\item reintentos;
\item deduplicación;
\item identificadores únicos;
\item registro durable del resultado lógico;
\item coordinación adicional.
\end{itemize}

\begin{uawarn}
En muchos sistemas reales, ``exactly-once'' no es una propiedad física simple de la red, sino una garantía lógica conseguida a costa de más estado, coordinación y complejidad.
\end{uawarn}

\section{Ejemplo trabajado: pago remoto}

Consideremos una operación \texttt{charge(requestId, amount)}.

\subsection{Escenario A: sin requestId}

\begin{enumerate}
\item el cliente envía cobro por \$100;
\item el servidor debita correctamente;
\item la respuesta se pierde;
\item el cliente expira;
\item el cliente reintenta;
\item el servidor interpreta el segundo request como uno nuevo;
\item se produce doble débito.
\end{enumerate}

\subsection{Escenario B: con requestId único}

\begin{enumerate}
\item el cliente envía \texttt{requestId=abc123};
\item el servidor procesa y registra que \texttt{abc123} ya produjo un débito;
\item la respuesta se pierde;
\item el cliente reintenta con el mismo \texttt{requestId};
\item el servidor detecta repetición y devuelve el resultado previo.
\end{enumerate}

\subsection{Conclusión}

La garantía deseada no surge del retry solo, sino del acoplamiento entre:

\begin{itemize}
\item retry;
\item request ID;
\item almacenamiento de resultados;
\item política de deduplicación.
\end{itemize}

\section{Backoff}

Los retries no deberían hacerse con intervalo fijo e inmediato. Una política de \emph{backoff} aumenta el tiempo entre reintentos, típicamente de forma exponencial.

Si $d_0$ es el delay inicial, un esquema clásico es:

\[
d_k = d_0 \cdot 2^k
\]

para el reintento $k$.

A menudo se agrega \emph{jitter}, es decir, aleatoriedad controlada, para evitar sincronización de clientes.

\subsection{Por qué importa}

Sin backoff:
\begin{itemize}
\item muchos clientes reintentan simultáneamente;
\item se agrava la sobrecarga;
\item se crean cascadas de falla.
\end{itemize}

Con backoff:
\begin{itemize}
\item se amortigua la presión sobre el sistema;
\item aumenta la probabilidad de recuperación;
\item se distribuye mejor la demanda en el tiempo.
\end{itemize}

\section{Qué debe saber hacer el estudiante al terminar esta clase}

El estudiante debería poder:

\begin{itemize}
\item explicar por qué una llamada remota no es semánticamente equivalente a una local;
\item justificar por qué un timeout no identifica por sí mismo la causa de una falla;
\item distinguir retry como mecanismo útil y como fuente de nuevos riesgos;
\item definir idempotencia formal e intuitivamente;
\item comparar at-most-once, at-least-once y exactly-once;
\item diseñar conceptualmente una operación remota segura ante reintentos.
\end{itemize}

\section{Errores conceptuales frecuentes}

\begin{itemize}
\item creer que ``sin respuesta'' equivale a ``no ocurrió'';
\item usar retry sin razonar sobre efectos duplicados;
\item confundir idempotencia con inmutabilidad;
\item creer que exactly-once es una propiedad trivial de la red;
\item fijar timeouts sin relación con dominio, latencia ni costo de error.
\end{itemize}

\end{itemize}

\section{Resumen final}

\begin{uakeybox}
La comunicación distribuida no consiste en mover bytes solamente, sino en decidir qué creer sobre operaciones remotas cuando la observación es incompleta. Timeouts, retries e idempotencia no son trucos de implementación: son respuestas parciales a una incertidumbre estructural.
\end{uakeybox}
