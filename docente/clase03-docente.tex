\section{Objetivo pedagógico profundo}
Mostrar que replicar estado no es copiar gratis: cada réplica adicional mejora disponibilidad o latencia, pero obliga a decidir qué divergencias temporales se aceptan y cómo se comunican al usuario.

\section{Resultados de aprendizaje esperados}
\begin{itemize}
\item explicar por qué se replica estado;
\item describir el patrón leader-follower;
\item comparar replicación síncrona y asíncrona;
\item reconocer stale reads y sus síntomas;
\item razonar sobre caída del líder y durabilidad de escrituras confirmadas.
\end{itemize}

\section{Guion sugerido}
\begin{enumerate}
\item Empezar con la motivación: disponibilidad, latencia y escala de lectura.
\item Dibujar líder y followers, separando escrituras de lecturas.
\item Contrastar sync y async con una escritura concreta.
\item Trabajar un stale read, por ejemplo cambio de contraseña o saldo.
\item Cerrar con la pregunta: ¿cuándo una escritura confirmada es durable?
\end{enumerate}

\section{Errores conceptuales frecuentes}
\begin{itemize}
\item pensar que más réplicas siempre implican más corrección;
\item olvidar que leer de followers puede devolver datos viejos;
\item confundir baja latencia con garantía fuerte;
\item no distinguir confirmación al cliente de propagación a réplicas.
\end{itemize}

\section{Preguntas disparadoras}
\begin{itemize}
\item ¿Desde dónde lee el cliente después de escribir?
\item ¿Qué pasa si el líder cae justo después de confirmar?
\item ¿Qué métrica operativa revela atraso de réplica?
\item ¿Qué preferimos para este dominio: frescura o disponibilidad?
\end{itemize}

Criterio de logro docente: la clase salió bien si el estudiante puede explicar una stale read sin llamarla simplemente ``bug''.
\end{uawarn}
