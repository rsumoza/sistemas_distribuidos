\section{Objetivo pedagógico profundo}
Que el estudiante entienda consistencia como una especificación observable de lecturas y escrituras, no como una palabra genérica para ``datos correctos''. La clase debe conectar modelos de consistencia con costo de coordinación, latencia y disponibilidad.

\section{Resultados de aprendizaje esperados}
\begin{itemize}
\item definir corrección en términos de historias observables;
\item explicar linearizability como comportamiento de objeto único;
\item explicar eventual consistency como convergencia con lecturas temporalmente obsoletas;
\item clasificar casos de dominio según el modelo de consistencia requerido;
\item relacionar consistencia fuerte con decisiones bajo partición.
\end{itemize}

\section{Guion sugerido}
\begin{enumerate}
\item Abrir con la pregunta: ``¿qué significa que el sistema sea correcto?''.
\item Presentar una escritura confirmada seguida de una lectura vieja.
\item Usar ese caso para distinguir linearizability de modelos más débiles.
\item Mostrar dominios donde la consistencia eventual es aceptable y dominios donde no.
\item Cerrar con el trade-off latencia--corrección y la conexión con CAP.
\end{enumerate}

\section{Errores conceptuales frecuentes}
\begin{itemize}
\item usar ``consistente'' sin especificar modelo;
\item creer que eventual consistency significa ausencia de garantías;
\item olvidar que linearizability respeta tiempo real entre operaciones no solapadas;
\item analizar consistencia sin mirar el dominio.
\end{itemize}

\section{Preguntas disparadoras}
\begin{itemize}
\item ¿Qué lectura esperaría un usuario después de una escritura confirmada?
\item ¿Qué historia observable permite este modelo?
\item ¿Qué costo tiene coordinar para preservar esa garantía?
\item ¿Qué daño produce una lectura vieja en este dominio?
\end{itemize}

\begin{uawarn}
Criterio de logro docente: la clase salió bien si el estudiante deja de decir ``consistencia'' en abstracto y nombra la garantía observable.
\end{uawarn}
