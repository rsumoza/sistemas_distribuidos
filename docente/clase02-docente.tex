\section{Objetivo pedagógico profundo}
Que el estudiante deje de pensar una llamada remota como una llamada local. La clase debe instalar que timeout, retry e idempotencia son decisiones de diseño bajo incertidumbre, no detalles de implementación.

\section{Resultados de aprendizaje esperados}
\begin{itemize}
\item distinguir falta de respuesta de falla comprobada;
\item explicar por qué RPC oculta la red pero no elimina sus problemas;
\item analizar retries en términos de recuperación y duplicación;
\item diseñar operaciones idempotentes mediante claves de idempotencia;
\item comparar at-most-once, at-least-once y exactly-once como garantías observables.
\end{itemize}

\section{Guion sugerido}
\begin{enumerate}
\item Abrir con una llamada a pagos que no responde.
\item Listar causas posibles antes de nombrar timeout.
\item Mostrar que retry recupera algunas fallas y crea riesgo de duplicación.
\item Introducir idempotencia como técnica para hacer seguro el retry.
\item Cerrar con semánticas de entrega y el trade-off confiabilidad--complejidad.
\end{enumerate}

\section{Errores conceptuales frecuentes}
\begin{itemize}
\item interpretar timeout como prueba de crash;
\item asumir que retry siempre mejora el sistema;
\item creer que idempotencia significa ``la operación no hace nada'';
\item vender exactly-once sin explicar deduplicación, persistencia y transacciones.
\end{itemize}

\section{Preguntas disparadoras}
\begin{itemize}
\item ¿Qué sabe el cliente cuando no recibe respuesta?
\item ¿Qué puede haber pasado en el servidor aunque el cliente observe timeout?
\item ¿Qué operación es segura de repetir y cuál no?
\item ¿Dónde se guarda la evidencia para deduplicar?
\end{itemize}

\begin{uawarn}
Criterio de logro docente: la clase salió bien si el estudiante puede justificar cuándo reintentar, cuándo no y cómo hacer que un retry no duplique efectos.
\end{uawarn}
