\section{Indicaciones generales}
En cada ejercicio indicar qué se gana con replicar y qué problema de consistencia, latencia o disponibilidad aparece como costo.

\section{Ejercicios}
\begin{uaexercise}\textbf{1.} Explicar por qué replicar estado mejora disponibilidad y latencia, pero introduce el problema de inconsistencia entre copias.\end{uaexercise}
\begin{uaexercise}\textbf{2.} Dibujar un esquema leader-follower con un líder y dos seguidores. Indicar qué nodo recibe escrituras, qué nodos podrían recibir lecturas y dónde aparece atraso de replicación.\end{uaexercise}
\begin{uaexercise}\textbf{3.} Comparar replicación síncrona y asíncrona para una escritura. Indicar cuál reduce stale reads y cuál mejora latencia de respuesta.\end{uaexercise}
\begin{uaexercise}\textbf{4.} Un usuario cambia su contraseña y luego intenta iniciar sesión contra una réplica atrasada. Explicar el síntoma, la causa y la garantía que faltó.\end{uaexercise}
\begin{uaexercise}\textbf{5.} Analizar qué puede pasar si el líder confirma una escritura, cae antes de replicarla y luego se promueve un follower.\end{uaexercise}
\begin{uaexercise}\textbf{6.} Elegir una política de lectura para un sistema de catálogo y para un sistema bancario: leer del líder o permitir followers. Justificar el trade-off.\end{uaexercise}
\begin{uaexercise}\textbf{7.} Definir una métrica o señal operativa para detectar stale reads o atraso de réplicas.\end{uaexercise}
\begin{uaexercise}\textbf{8.} Aplicar problema--fallas--garantía--solución--trade-off a un sistema replicado que debe tolerar caída de un nodo.\end{uaexercise}
