\section{Indicaciones generales}
En cada ejercicio indicar qué se gana con replicar y qué problema de consistencia, latencia o disponibilidad aparece como costo.

\section{Ejercicios y respuestas orientativas}
\begin{uaexercise}\textbf{1.} Explicar por qué replicar estado mejora disponibilidad y latencia, pero introduce el problema de inconsistencia entre copias.\end{uaexercise}
\begin{uaanswer}Mejora disponibilidad porque otra copia puede responder si una falla, y mejora latencia si se ubican réplicas cerca de los usuarios. El costo es que las copias pueden no estar en el mismo estado al mismo tiempo, especialmente con propagación asíncrona.\end{uaanswer}
\begin{uaexercise}\textbf{2.} Dibujar un esquema leader-follower con un líder y dos seguidores. Indicar qué nodo recibe escrituras, qué nodos podrían recibir lecturas y dónde aparece atraso de replicación.\end{uaexercise}
\begin{uaanswer}El líder recibe escrituras y propaga cambios a followers. Si las lecturas van a followers, pueden ver estado atrasado. El atraso aparece entre la confirmación en el líder y la aplicación del cambio en cada follower.\end{uaanswer}
\begin{uaexercise}\textbf{3.} Comparar replicación síncrona y asíncrona para una escritura. Indicar cuál reduce stale reads y cuál mejora latencia de respuesta.\end{uaexercise}
\begin{uaanswer}La síncrona espera confirmación de réplicas y reduce divergencia, pero aumenta latencia y puede bajar disponibilidad. La asíncrona responde antes y mejora latencia, pero permite stale reads y pérdida de escrituras recientes si el líder cae.\end{uaanswer}
\begin{uaexercise}\textbf{4.} Un usuario cambia su contraseña y luego intenta iniciar sesión contra una réplica atrasada. Explicar el síntoma, la causa y la garantía que faltó.\end{uaexercise}
\begin{uaanswer}Síntoma: el sistema confirmó el cambio, pero luego rechaza la nueva contraseña. Causa: la lectura/autenticación fue contra un follower sin la actualización. Faltó una garantía de read-your-writes o una política de lectura consistente después de una escritura.\end{uaanswer}
\begin{uaexercise}\textbf{5.} Analizar qué puede pasar si el líder confirma una escritura, cae antes de replicarla y luego se promueve un follower.\end{uaexercise}
\begin{uaanswer}El nuevo líder puede no tener la escritura confirmada. Desde el cliente, la operación parecía exitosa; desde el sistema, no quedó durable en réplicas suficientes. Esto obliga a definir cuándo una escritura se considera realmente confirmada y durable.\end{uaanswer}
\begin{uaexercise}\textbf{6.} Elegir una política de lectura para un sistema de catálogo y para un sistema bancario: leer del líder o permitir followers. Justificar el trade-off.\end{uaexercise}
\begin{uaanswer}Un catálogo suele tolerar followers para mejorar latencia y escala, aceptando desactualización temporal. Un sistema bancario suele preferir líder o quórum para preservar corrección de saldos, aceptando más latencia o menor disponibilidad.\end{uaanswer}
\begin{uaexercise}\textbf{7.} Definir una métrica o señal operativa para detectar stale reads o atraso de réplicas.\end{uaexercise}
\begin{uaanswer}Se puede medir replication lag por tiempo o por posición de log, comparar versión leída contra versión confirmada, registrar read-after-write failures o monitorear edad máxima de datos por réplica.\end{uaanswer}
\begin{uaexercise}\textbf{8.} Aplicar problema--fallas--garantía--solución--trade-off a un sistema replicado que debe tolerar caída de un nodo.\end{uaexercise}
\begin{uaanswer}Problema: mantener servicio ante caída de una réplica. Fallas: crash de follower o líder, atraso de red. Garantía: lecturas disponibles con frescura acotada o escrituras durables en mayoría. Solución: leader-follower con confirmación según política. Trade-off: latencia y complejidad contra disponibilidad y consistencia.\end{uaanswer}
