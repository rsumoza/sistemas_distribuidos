\section{Resoluciones}

\begin{uaexercise}
¿Por qué WAL garantiza atomicidad?
\end{uaexercise}

\begin{uaanswer}
Porque toda operación se registra antes de ejecutarse. Si ocurre un crash, el sistema puede:
\begin{itemize}
\item aplicar REDO para operaciones committed
\item aplicar UNDO para operaciones incompletas
\end{itemize}
Esto asegura que la transacción se ejecute completamente o no se ejecute.
\end{uaanswer}

\begin{uaexercise}
Ejemplo de MVCC.
\end{uaexercise}

\begin{uaanswer}
Versiones:
\[
(x,10,t=1), (x,20,t=5), (x,30,t=10)
\]

Transacción con $t=6$:
\[
\Rightarrow x=20
\]

Se selecciona la versión más reciente válida.
\end{uaanswer}

---

\begin{uaexercise}
Write skew en MVCC.
\end{uaexercise}

\begin{uaanswer}
Dos transacciones leen mismo estado inicial y escriben sobre datos distintos sin conflicto directo.

No hay conflicto RW ni WW ⇒ ambas committean.

Resultado: inconsistencia lógica.
\end{uaanswer}

---

\begin{uaexercise}
LSM escritura paso a paso.
\end{uaexercise}

\begin{uaanswer}
\begin{enumerate}
\item escribir en WAL
\item insertar en Memtable
\item flush a SSTable
\item compaction
\end{enumerate}
\end{uaanswer}

---

\begin{uaexercise}
Write amplification.
\end{uaexercise}

\begin{uaanswer}
Ocurre porque un mismo dato se reescribe múltiples veces durante compaction.
Esto incrementa el costo total de escritura.
\end{uaanswer}
