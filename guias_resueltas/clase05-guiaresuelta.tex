\begin{uaexercise}
Construir write skew.
\end{uaexercise}

\begin{uaanswer}
Supongamos dos cuentas A y B, con la regla: al menos una debe tener saldo positivo.

Estado inicial:
\[
A=10,\quad B=10
\]

Transacción \(T_1\):
\begin{itemize}
\item lee \(A=10, B=10\)
\item escribe \(A=0\)
\end{itemize}

Transacción \(T_2\):
\begin{itemize}
\item lee \(A=10, B=10\)
\item escribe \(B=0\)
\end{itemize}

Bajo Snapshot Isolation, ambas pueden confirmar porque no escriben el mismo dato.

Estado final:
\[
A=0,\quad B=0
\]

Se viola la restricción global.

Este es el fenómeno de \textbf{write skew}: cada transacción es válida en aislamiento, pero la ejecución concurrente no es equivalente a ninguna ejecución serial válida.
\end{uaanswer}

\begin{uaexercise}
2PC con falla.
\end{uaexercise}

\begin{uaanswer}
En Two-Phase Commit:

\begin{itemize}
\item el coordinador solicita \emph{prepare}
\item los participantes responden YES
\end{itemize}

Si el coordinador falla antes de enviar COMMIT:

\begin{itemize}
\item los participantes quedan en estado \emph{prepared}
\item no pueden decidir commit ni abort por sí solos
\end{itemize}

Resultado:

\begin{quote}
\textbf{bloqueo indefinido}
\end{quote}

Esto muestra que 2PC garantiza atomicidad, pero no disponibilidad.
\end{uaanswer}
